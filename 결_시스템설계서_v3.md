# 결 (Gyeol) — 시스템 설계서 (v3)

> 본 문서는 v2에 (1) 매트릭스 엔진 v3 + 매칭 알고리즘 v7 반영, (2) 비용 추정 정정 (~$720/월 10K 사용자), (3) PostgreSQL 스키마 갱신 (polish_cache_key 분리, polished_output_cache 별도 테이블, evidence_quote 폐기), (4) Apple Speech Framework 음성 입력 통합, (5) PIPA 동의서에 음성 데이터 처리 명시를 반영한 v3.

> v2의 핵심 결정사항(iOS 네이티브, Supabase Seoul, Apple Sign In, Vertex AI, Gemini 3 Flash + 3.1 Flash-Lite 분리)은 그대로 유지.

> **v2 → v3 주요 변경**: §15 참조.

## 2. 시스템 아키텍처

### 2.1 전체 구조 (v3 갱신)

```
┌─────────────────────────────────────┐
│  iOS App (Swift + SwiftUI)          │
│                                      │
│  + Speech Framework (음성 입력)      │ [v3 신설]
│  + AuthenticationServices (Apple)    │
└─────────────────────────────────────┘
              ↕ HTTPS (TLS 1.3)
              ↕ Supabase Swift SDK
┌─────────────────────────────────────┐
│  Supabase (Seoul ap-northeast-2)    │
│                                      │
│  - Auth (Apple Sign In)              │
│  - PostgreSQL                        │
│    + polished_output_cache (v3)      │
│    + boundary_check_payload (v3)     │
│  - Realtime                          │
│  - Edge Functions (Deno)             │
│    - llm-prompt-a                    │
│    - llm-prompt-b                    │
│    - llm-prompt-c-postedit           │
│    - llm-prompt-d                    │
│    - llm-prompt-e                    │
│    - matching-algorithm              │
│    - explanation-payload-builder     │ [v3 신설]
│    - recommendation-matrix-engine    │
│    - post-polish-validation          │ [v3 신설]
│    - normalization-worker            │
│    - raw-quote-detector              │ [v3 신설]
└─────────────────────────────────────┘
              ↕ HTTPS
┌─────────────────────────────────────┐
│  Vertex AI (asia-northeast3 Seoul)  │
│  - Gemini 3 Flash (A·B·D)            │
│  - Gemini 3.1 Flash-Lite (C·E)       │
└─────────────────────────────────────┘
```

**v3 신설 Edge Functions**:
- `explanation-payload-builder`: 최종 큐 후보에만 호출. pair_reason_atoms, public sentences, boundary_check_payload 산출
- `post-polish-validation`: 후편집 LLM 결과 8가지 검증
- `raw-quote-detector`: 정규화 레이어에서 summary의 raw quote 감지

### 2.2 데이터 흐름 (v3 갱신)

**(1) 인터뷰 진행 → 영역 분석 → 핵심 유형 → 명시 dealbreaker**:

기본 흐름은 v2 그대로 + 다음 추가:
- **음성 입력 옵션**: 사용자가 마이크 버튼 탭 시 Speech Framework 호출 → 변환된 텍스트가 답변 입력 영역에 자동 채워짐
- 음성은 *기기 내에서만* 처리됨. 텍스트만 백엔드 전송.

**(2) 매칭 흐름 (v3 갱신)**:

```
사용자 [이대로 발행하기]
  ↓
matching-algorithm Edge Function
  - System Hard / User Hard 필터링
  - 호환 점수 계산
  - alignment_by_domain 산출 (alignment_level만)
  - compatibility_assessment_basic 완성
  ↓
matches 테이블 저장 (basic만, explanation_payload는 NULL)
  ↓
[2단계: 점수 정렬 + 필터 통과 후보 산출]
  ↓
[3단계: 최종 큐 후보 산출 — UI 노출용]
  ↓ Edge Function trigger (비동기)
explanation-payload-builder Edge Function (페어별)
  - pair_reason_atoms 산출
  - public_alignment_sentence / public_tension_sentence 생성
  - boundary_check_payload 산출 (해당 페어)
  - matches 테이블에 explanation_payload 채움
  ↓ Edge Function trigger
recommendation-matrix-engine Edge Function (페어별)
  - draft_narrative 결정론적 조립
  - candidate_brief 구성 (pair relevance 정렬)
  - 후편집 평가 함수
  ↓
needs_polish == true ?
  ├─ Yes → polished_output_cache 캐시 확인
  │         ├─ hit → 캐시 결과 + post-polish-validation
  │         └─ miss → llm-prompt-c-postedit (Vertex AI)
  │                   ↓
  │                 post-polish-validation (8가지 검증)
  │                   ├─ pass → 캐시 저장 + 사용
  │                   └─ fail → draft 폴백
  └─ No → draft_narrative 그대로
  ↓
matches 테이블 갱신 (recommendation_narrative)
  ↓
사용자 매칭 화면 진입 시 목록 노출
```

**v3 핵심 변경**:
- 매칭 알고리즘이 *최종 큐 후보 산출까지*만 수행. explanation_payload는 별도 Edge Function
- `polished_output_cache`로 후편집 결과 캐싱
- post-polish-validation Edge Function 신설

## 3. 데이터 모델

### 3.1 PostgreSQL 스키마 (v3 갱신)

v2 §3.1 그대로 + 다음 변경:

#### 3.1.1 `analyses` 테이블 — evidence_quote 폐기 (v3)

```sql
ALTER TABLE analyses DROP COLUMN structured_evidence_quote;
-- v6에서 structured.principle_mix.evidence_quote가 JSONB에 저장되었으나
-- v7부터 폐기. 대신 evidence_ids만 structured에 남고
-- 실제 quote는 answer_evidence 테이블에 별도 보관

CREATE TABLE answer_evidence (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    analysis_id UUID NOT NULL REFERENCES analyses(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    domain_id TEXT NOT NULL,
    
    evidence_id TEXT NOT NULL, -- ev_001, ev_002 ...
    quote TEXT NOT NULL, -- raw 인용 (self_review_only)
    context TEXT,
    
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_answer_evidence_user ON answer_evidence(user_id);
```

본인 검토 화면에서만 접근. 매칭 알고리즘과 매트릭스 엔진은 evidence_ids만 사용.

#### 3.1.3 `polished_output_cache` 테이블 (v3 신설)

후편집 LLM 결과 캐싱.

```sql
CREATE TABLE polished_output_cache (
    cache_key TEXT PRIMARY KEY, -- SHA-256 hash
    
    viewer_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    candidate_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    viewer_profile_version TEXT NOT NULL,
    candidate_profile_version TEXT NOT NULL,
    assessment_version TEXT NOT NULL,
    template_library_version TEXT NOT NULL,
    polish_prompt_version TEXT NOT NULL,
    
    draft_hash TEXT NOT NULL,
    
    polished_headline TEXT NOT NULL,
    polished_alignment_narrative TEXT NOT NULL,
    polished_tension_narrative TEXT NOT NULL,
    
    validation_passed BOOLEAN NOT NULL,
    validation_failure_reason TEXT, -- raw_quote_introduced, evaluative_word_introduced 등
    
    cached_at TIMESTAMPTZ DEFAULT NOW(),
    expires_at TIMESTAMPTZ -- 90일 또는 polish_prompt_version 변경 시
);

CREATE INDEX idx_polished_cache_viewer ON polished_output_cache(viewer_id);
CREATE INDEX idx_polished_cache_expires ON polished_output_cache(expires_at);
```

캐시 key에 viewer_id 포함 → A→B와 B→A 분리.

### 4.6 음성 입력 통합 (v3 신설)

#### 4.6.1 SDK 선택

**Apple Speech Framework (SFSpeechRecognizer)** + iOS 시스템 받아쓰기.

근거:
- iOS 10+ 표준 SDK, 한국어 (ko-KR) 지원, 무료
- on-device 모드 (iOS 13+, A9 이상): 외부 서버 전송 없음
- 외부 STT API (Google Cloud, Whisper) 거부 — 비용 + 민감 정보 외부 전송 부담

#### 4.6.2 통합 방식

**iOS App 측**:
```swift
import Speech

class VoiceInputManager: ObservableObject {
    private let speechRecognizer = SFSpeechRecognizer(locale: Locale(identifier: "ko-KR"))
    private var audioEngine = AVAudioEngine()
    private var recognitionTask: SFSpeechRecognitionTask?
    
    @Published var transcribedText: String = ""
    @Published var isRecording: Bool = false
    
    // on-device 모드 강제
    private func startRecording() {
        let request = SFSpeechAudioBufferRecognitionRequest()
        request.requiresOnDeviceRecognition = true
        // ... 60초 무음 감지, 1분 제한 우회 (자동 재시작)
    }
}
```

**1분 제한 우회**:
- 1분 가까워지면 (50초 시점) 자동 세션 재시작
- 누적 텍스트 유지
- 사용자에게는 연속 녹음으로 보임

**60초 무음 자동 종료**:
- AVAudioEngine으로 입력 레벨 모니터링
- 60초 무음 감지 시 자동 종료
- 토스트 안내

#### 4.6.3 백엔드 통합

음성 입력 자체는 *백엔드 통합 없음*. 변환된 텍스트만 일반 답변 텍스트로 백엔드 전송.

운영 분석:
- `interviews.voice_input_used`: 사용자가 영역 인터뷰에서 음성 사용 여부
- `interviews.voice_input_session_count`: 누적 음성 세션 수

## 5. 매칭 알고리즘 구현 (v3 갱신)

### 5.1 매칭 풀 계산 전략 (v3 갱신)

**3단계 분리**:

1. **전체 페어 호환 점수 계산** (배치 1시간):
   - System Hard / User Hard 필터링
   - 호환 점수 + alignment_level
   - compatibility_assessment_basic
   - matches 테이블에 basic만 저장 (explanation_payload는 NULL)

2. **최종 큐 후보 산출** (사용자 매칭 화면 진입 시):
   - 사용자별 점수 정렬
   - 노출 후보 30명 산출

3. **explanation_payload + 매트릭스 엔진** (비동기 큐):
   - 최종 큐 후보에 대해서만 explanation_payload 산출
   - 매트릭스 엔진 호출
   - 후편집 (필요 시) + post-polish-validation
   - matches 테이블에 explanation_payload + recommendation_narrative 저장

부하 비교:
- v2: 50M 페어 모두 atoms 산출 → 큰 부담
- v3: 1단계는 50M (basic만, ~5-10ms/페어), 3단계는 90K/일 (atoms + narrative, ~25ms/페어)

### 5.2 Edge Function 호출 의사코드 (v3)

```typescript
// 1단계: 전체 페어 — matching-algorithm
async function computeBasicAssessment(pairs: Pair[]) {
    for (const pair of pairs) {
        const basic = await computeCompatibilityBasic(pair);
        await db.matches.upsert({
            viewer: pair.a, candidate: pair.b,
            compatibility_assessment_basic: basic
        });
    }
}

// 3단계: 최종 큐 후보 — explanation-payload-builder + matrix engine
async function generateRecommendation(matchId: UUID) {
    const match = await db.matches.get(matchId);
    
    // explanation_payload 산출
    const payload = await buildExplanationPayload(match);
    await db.matches.update(matchId, { explanation_payload: payload });
    
    // boundary_check_payload (해당 페어)
    if (match.compatibility_assessment_basic.qualitative_label === '경계 확인') {
        const boundary = await buildBoundaryCheckPayload(match);
        await db.matches.update(matchId, { boundary_check_payload: boundary });
    }
    
    // 매트릭스 엔진
    const draft = await callMatrixEngine(match);
    
    // 후편집 평가
    const needsPolish = evaluateDraftQuality(draft);
    
    let finalNarrative = draft;
    let polishApplied = false;
    let validationPassed = true;
    
    if (needsPolish) {
        // 캐시 확인
        const cacheKey = computePolishCacheKey(match, draft);
        const cached = await db.polishedOutputCache.get(cacheKey);
        
        let polished;
        if (cached) {
            polished = cached;
        } else {
            polished = await callLLMPostedit(draft);
        }
        
        // post-polish-validation
        const validation = validatePolishOutput(draft, polished, match);
        
        if (validation.valid) {
            finalNarrative = polished;
            polishApplied = true;
            
            // 캐시 저장
            if (!cached) {
                await db.polishedOutputCache.insert({
                    cache_key: cacheKey,
                    polished,
                    validation_passed: true
                });
            }
        } else {
            // 검증 실패 → draft 폴백
            finalNarrative = draft;
            polishApplied = false;
            validationPassed = false;
            
            // 캐시 저장 안 함 (실패한 polish 결과 재사용 금지)
            
            log.warn('polish_validation_failed', validation.reason);
        }
    }
    
    // matches 테이블 갱신
    await db.matches.update(matchId, {
        recommendation_narrative: finalNarrative,
        polish_applied: polishApplied,
        polish_validation_passed: validationPassed,
        recommendation_status: 'ready'
    });
}
```
